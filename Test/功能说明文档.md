## 各文件详细分析

### 1. `app.py` — Gradio 主应用入口

**实现原理：**
- 使用 `gradio.Blocks` 构建 3 个 Tab 页（预测总览、数据探索、深度分析）
- 通过 `model_service` 执行预测，通过 `weather_service` 获取气象数据
- 使用 Plotly 渲染图表（负荷曲线、Pearson 相关性热力图、小时画像、回测散点图）
- 配置了 CSS 样式（仪表板卡片、深色模式切换）

**真实可用部分：**
- ✅ Gradio UI 框架 — 布局、Tab、按钮绑定、事件回调完全可用
- ✅ 数据探索 Tab — 直接读取 `Data/` 目录下 CSV 文件
  - 数据集总览：显示形状、列名、基础统计（describe）
  - 相关性分析：`df.corr()` 计算 Pearson 系数，Plotly 热力图渲染
  - 小时负荷画像：`groupby('hour')['load_kw'].mean()` 聚合，分训练/测试两色绘制
- ✅ 气象 Tab — 调用 `weather_service.update_weather_ui()`，走真实 Open-Meteo API
- ✅ 深度分析 Tab — `error_analysis_backtest()` 使用 `sklearn.neural_network.MLPRegressor` 做代理回测
- ✅ 深色模式切换 — 纯前端 JavaScript `document.querySelector('body').classList.toggle('dark')`

**不可用/占位部分：**
- ❌ 预测核心 Tab — `model_service.run_prediction()` 走**完全模拟**路径：
  ```python
  # 光伏：正弦曲线模拟
  solar = np.maximum(0, np.sin(2 * np.pi * hours / 24 - np.pi/6) * 100 + noise)
  # 充电：正态分布随机模拟
  load = 30 + 20 * np.sin(2 * np.pi * hours / 24 - np.pi/4) + np.random.normal(0, 10, n_steps)
  ```
- ❌ 充电预测策略建议为空（只有标题）

**结论：** `app.py` 是 **Demo 原型**。UI 完整、交互流畅，但预测结果全部来自数学模拟。适合演示概念流程，不适合作为实际预测工具。

### 3. `model_service.py` — 预测模型服务（核心文件）

**实现原理：**

此文件定义了多个类和服务：

#### 3.1 `SolarPredictor`（62-198 行）
- 尝试从 `Solar_Forecast/best_pth/best_generator.pth` 加载真实权重
- **尝试加载的模型类**：`CNN_LSTM` + `GeneratorWithFeatures`（从 `Solar_Forecast/NN.py` 导入）
- **参数配置**：`input_size=5, hidden_size=64, num_layers=2, cnn_channels=[64,64]`
- **必定失败原因**：权重文件 `best_generator.pth` 是 `LSTMPredictor`（由 `forecast_LSTM_GAN.ipynb` 保存），而非 `CNN_LSTM`。`load_state_dict()` 会因 key 不匹配抛出异常，然后回退到 `_mock_predict()`
- **`_build_features()` 方法**：用天气单值重复填充 96 步序列，与训练时的 7 维 MinMaxScaler 特征完全不匹配
- **`_mock_predict()` 回退**：正弦曲线 + 随机噪声模拟

#### 3.2 `ChargingLoadPredictor`（201-359 行）
- 尝试从 `Charging_Forecast/best_pth/final_best_hybrid_model.pth` 加载真实权重
- **尝试加载的模型类**：`CNN_LSTM_Advanced`（从 `NN.py` 导入）
- **参数配置**：`input_size=6, hidden_size=64, cnn_channels=[64,128], use_attention=True`
- **必定失败原因**：权重文件是 `HybridModel`（TCN-Attention-LSTM），而非 `CNN_LSTM_Advanced`。key 完全不匹配
- **`_build_features()` 方法**：从历史 CSV 取最近 96 行原始值，**未经过 MinMaxScaler 归一化**（训练时数据是 scaler_X 归一化后的）
- **`_mock_predict()` 回退**：分时段模拟曲线（早晚高峰）

#### 3.3 `PredictionService`（362-404 行）
- 组合 `SolarPredictor` + `ChargingLoadPredictor`
- 计算净负荷、绿电替代率、节省电费
- 计算逻辑本身正确，但**输入数据全为模拟**

#### 3.4 `AlertService`（407-526 行）
- 策略建议算法正确：检查超阈值负荷、光伏盈余、傍晚无光
- 按小时聚合净负荷，生成分时充放电建议
- 逻辑真实可用，但**输入数据全为模拟**

#### 3.5 Adapter 层（539-660 行）
- `SolarForecastServiceAdapter` / `ChargingForecastServiceAdapter` 包装为 Gradio 统一接口
- 全局实例：`solar_forecast_service`、`charging_forecast_service`

**真实可用性：⚠️ 0%（预测核心）/ ✅ 部分（策略算法）**

| 组件 | 真实可用 | 原因 |
|------|----------|------|
| 光伏模型加载 | ❌ | 模型类不匹配，100% 回退到模拟 |
| 充电模型加载 | ❌ | 模型类不匹配，100% 回退到模拟 |
| 特征构建 | ❌ | 未使用 MinMaxScaler，数值量纲错误 |
| 模拟回退函数 | ❌ | 纯数学公式，非模型推理 |
| 策略/预警算法 | ✅ | 逻辑正确，可独立使用 |
| Adapter 接口 | ✅ | 接口设计良好，数据管道通畅 |

### 6. `weather_service.py` — 气象数据服务

**实现原理：**
- `fetch_current_weather()`：HTTP GET → Open-Meteo `/v1/forecast`，获取当前辐照度/云量/降雨/温度/风速
- `fetch_weather_data()`：获取实时 + 15 分钟分辨率预报数据，对齐时区到 `Asia/Shanghai`
- `create_radiation_plot()`：Plotly 双 Y 轴图（辐射强度 + 云量曲线）
- `update_weather_ui()`：Gradio 组件刷新入口，返回图 + 预警 HTML + DataFrame + 实时信息
- `get_weather_data()`：为模型预测准备天气输入字典

**真实可用性：✅ 100%**
- 真实 HTTP 请求，返回 Open-Meteo 免费预报数据
- 数据处理链条完整（raw JSON → parsed dict → DataFrame → Plotly chart）
- 模拟回退机制完整（API 不可用时自动降级）
- 唯一依赖：需要网络连接
- `Test2/` 版本在此基础上进行了重构

## 关键问题汇总

### 问题 1：模型类不匹配（致命缺陷）

```
model_service.py 使用的类      vs    权重文件实际保存的类
─────────────────────────────────────────────────────────
CNN_LSTM (input_size=5)       vs    LSTMPredictor (input_size=7)
CNN_LSTM_Advanced             vs    HybridModel (TCN-Attention-LSTM)
```

**根因：** 代码使用了与训练 notebook 完全不同的模型架构，key 必定不匹配。

### 问题 2：特征构建与训练时不一致

```python
# model_service.py 中 _build_features()：
features[:, 0] = radiation   # 固定值填充
features[:, 1] = cloudcover  # 固定值填充
# ❌ 未使用 MinMaxScaler 归一化
# ❌ 使用了训练时不存在的 feature (cloudcover 替代 hour_sin)
```

### 问题 3：预测全走模拟路径

所有模型加载都会抛出异常（key mismatch → except → `_mock_predict()`）。用户运行时看到的始终是模拟数据，不会有任何真实预测结果。
