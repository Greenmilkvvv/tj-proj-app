## 1. 项目背景

### 2.2 `model_service.py` — 预测服务核心

**功能**：加载模型、执行光伏/充电预测、生成策略建议。

**实现原理**：
- **模型加载**：从 pth 文件加载 PyTorch state_dict，但采用了**推测式参数恢复**（通过读取权重张量的 shape 反推 hidden_size 等参数），而非从训练代码直接引用模型类
- **光伏预测**：加载 `best_generator.pth`，构建 `LSTMPredictor`（2层LSTM, hidden=128），用 `simple_scaler_1.pkl` 做归一化/逆归一化，从 `aligned_2026_01_02.csv` 取历史窗口
- **充电预测**：加载 `final_best_hybrid_model.pth`，构建 `HybridModel`（TCN-Attention-LSTM, hidden 从权重推断），用 `scaler_X.pkl`/`scaler_y.pkl` 做归一化，从 `dataset_selected_features_test.csv` 取历史窗口
- **迭代预测**：逐步预测并滚动更新窗口，光伏侧实时合成 `hour_sin`/`hour_cos` 和辐射特征，充电侧实时更新 `lag_1`/`lag_96`/`lag_672`/`rolling_std_4`/`rolling_mean_4`
- **降级策略**：当模型或 scaler 文件不存在时，自动回退到模拟预测（`_mock_prediction`），使用正弦曲线模拟光伏、高斯混合模拟负荷
- **不确定性**：充电负荷通过 ±15% 简单估算上下界

**可用性评估**：✅ **核心可用**。模型加载、预测、降级三条路径均完整实现。

## 3. 核心实现原理

### 3.1 预测流程

```
用户选择步数 → app.py 调用 model_service.run_prediction()
                      │
                      ├─ 尝试加载光伏模型 (best_generator.pth)
                      │   ├─ 成功 → _real_prediction → _predict_solar()
                      │   └─ 失败 → _mock_prediction (正弦模拟)
                      │
                      ├─ 尝试加载充电模型 (final_best_hybrid_model.pth)
                      │   ├─ 成功 → _real_prediction → _predict_charging()
                      │   └─ 失败 → _mock_prediction (高斯混合模拟)
                      │
                      └─ generate_strategy() → 策略文本
```

### 3.2 光伏预测模型架构

- **训练代码**：`Solar_Forecast/NN.py`
- **模型架构**：`LSTMPredictor`（2层 LSTM, hidden=128）→ 包装为 `GeneratorWithFeatures`（GAN 训练框架）
- **训练方式**：`GANTrainer` 对抗训练，联合 MSE + GAN loss
- **输入特征**（7维）：`power`, `hour_sin`, `hour_cos`, `shortwave_radiation`, `direct_radiation`, `diffuse_radiation`, `direct_normal_irradiance`
- **输入窗口**：24 步（6小时，15min/步）
- **输出**：1 步（下一时刻功率，kW）
- **归一化**：`MinMaxScaler`（joblib 格式）

### 3.3 充电预测模型架构

- **训练代码**：`Charging_Forecast/Bys-TCN-Attention-LSTM_model.ipynb`
- **模型架构**：`HybridModel` — TCN（5层膨胀卷积）→ SimpleAttention → LSTM → 全连接层
- **输入特征**（6维）：`price`, `lag_1`, `lag_96`, `lag_672`, `rolling_std_4`, `rolling_mean_4`
- **输入窗口**：96 步（24小时，15min/步）
- **输出**：1 步（下一时刻充电负荷，kW）
- **归一化**：`MinMaxScaler`（pickle 格式），X 和 y 分别归一化

### 3.4 迭代预测机制

由于模型为单步预测，多步预测通过**滚动迭代**实现：

1. 取历史窗口（光伏24步/充电96步）输入模型
2. 模型输出下一步预测值
3. 将预测值反归一化后作为新特征加入窗口
4. 滑动窗口丢弃最旧一步
5. 重复 1-4 直到达到目标步数

**光伏侧额外处理**：每一步需重新计算 `hour_sin`/`hour_cos`（基于时间推进15分钟），以及从天气 API 获取辐射特征。若无天气数据则沿用历史辐射值。

**充电侧额外处理**：每一步需维护 `pred_history` 队列以计算 `lag_1`/`lag_96`/`lag_672` 和 `rolling_std_4`/`rolling_mean_4`。

### 3.5 策略生成逻辑

基于光伏出力与充电负荷的差值（net = solar - load）：
- **净盈余 > 5kW** → 建议储能充电（绿电过剩）
- **净缺口 > 5kW** → 建议储能放电（补充缺口）
- **差值在 ±5kW 内** → 供需平衡，无需调度

## 6. 已知问题与风险

### 6.1 技术债务

1. **模型类定义重复**：`model_service.py` 中内联定义了 `TCNBlock`、`SimpleAttention`、`HybridModel` 等类，与训练 Notebook 中的定义重复，存在维护风险
2. **参数硬编码**：`_CHARGING_HIDDEN_SIZE = 121`、`_SOLAR_HIDDEN_SIZE = 128` 等参数硬编码，若重新训练模型可能不匹配
3. **Scaler 格式不一致**：光伏用 `joblib`，充电用 `pickle`，需同时安装两个库
4. **错误处理不完善**：部分异常仅 print 警告后静默降级，可能导致用户未察觉使用了模拟数据

### 6.2 运行时风险

1. **Gradio 版本兼容性**：`gr.LinePlot` 的 API 在较新版本中可能已变更
2. **迭代预测发散**：长时间预测（12小时=48步）时误差累积可能导致预测值严重偏离
3. **数据时效性**：`aligned_2026_01_02.csv` 包含截至 2026-01-02 的数据，若当前日期远超此日期，lag 特征的参考价值降低
4. **天气 API 限流**：Open-Meteo 免费 API 可能有请求频率限制
